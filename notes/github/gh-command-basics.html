ghコマンドの基本的な使い方

作成日: 2026-06-28

カテゴリ: github

元の質問

ghコマンドの基本的な使い方は？

回答

gh は GitHub CLI のコマンドで、リポジトリ、Issue、Pull Request、GitHub Actions などの GitHub 上の操作をブラウザではなく端末から扱うための道具です。公式マニュアルでは gh の下に auth、repo、issue、pr、run などのサブコマンドが並び、操作対象ごとに命令を組み立てる構造になっています。[1]

最初は「認証する」「今いるリポジトリを確認する」「Issue と PR を見る」「必要なときだけ作成する」の 4 つを覚えると十分です。破壊的ではない確認系コマンドから使い始め、作成系は --web や明示的な --title / --body つきで実行すると、意図しない投稿を避けやすくなります。

まず認証する

対話的にログインする基本形は gh auth login です。公式マニュアルでは、既定のホストは github.com、既定の認証はブラウザを使うフローで、完了後のトークンは通常システムの資格情報ストアに保存されると説明されています。[2]

コード例（言語名: shell）:

gh auth login
gh auth status

gh auth status は、既知の GitHub host ごとに有効なアカウントと認証状態を表示します。自動化では、標準入力からトークンを渡す --with-token や環境変数のトークンも使えます。ただし個人アクセストークンや fine-grained token は権限スコープによって挙動が変わるため、まずは手元の端末で gh auth login を使い、CI やスクリプトに移す段階でトークン運用を分けて考えるのが実務上は安全です。[2][3]

リポジトリを見る・開く・clone する

現在のディレクトリが GitHub リポジトリなら、引数なしの gh repo view でそのリポジトリの説明や README を端末に表示できます。別のリポジトリを指定する場合は OWNER/REPO 形式を使い、--web を付けるとブラウザで開けます。[4]

コード例（言語名: shell）:

gh repo view
gh repo view cli/cli
gh repo view cli/cli --web
gh repo clone cli/cli

gh repo clone OWNER/REPO は GitHub リポジトリをローカルに clone します。OWNER/ 部分を省略すると、認証中のユーザー名が owner として扱われます。また、clone 対象が fork の場合は親リポジトリが upstream remote として追加される挙動があります。[5]

Issue を見る・作る

gh issue list はリポジトリの Issue 一覧を表示します。既定では open な Issue だけが対象で、--state、--assignee、--author、--label、--search などで絞り込めます。[6]

コード例（言語名: shell）:

gh issue list
gh issue list --assignee "@me"
gh issue list --state all --label bug
gh issue view 123
gh issue create --title "調査する" --body "再現手順と確認事項を書く"

gh issue view は Issue 番号または URL を指定して、タイトル、本文、その他の情報を表示します。gh issue create は Issue を作成します。タイトルや本文をフラグで渡せるほか、--web を使うとブラウザの作成画面を開けます。プロジェクトへ追加する場合は追加の認可スコープが必要になることがあります。[7][8]

Pull Request を見る・作る

gh pr list は Pull Request 一覧を表示します。既定では open な PR が対象で、--state、--author、--assignee、--base、--head、--search などで絞り込めます。[9]

コード例（言語名: shell）:

gh pr list
gh pr list --author "@me"
gh pr view 123
gh pr checkout 123
gh pr create --title "修正内容" --body "変更点と確認方法を書く"
gh pr create --draft --fill

gh pr view は PR 番号、URL、またはブランチを指定して、タイトル、本文、その他の情報を表示します。gh pr create は PR を作成し、成功すると作成された PR の URL を表示します。現在のブランチが remote に完全には push されていない場合、どこへ push するかを尋ねたり fork を提案したりするため、作成前に git status と git branch --show-current で状態を確認しておくと混乱しにくいです。[10][11]

GitHub Actions を確認する

gh run list は最近の GitHub Actions workflow run を一覧表示します。--branch、--commit、--event、--status、--workflow などで絞り込めます。run の詳細は gh run view で確認でき、PR に紐づく CI 状態を見る用途では gh pr checks が使えます。[12][13][14]

コード例（言語名: shell）:

gh run list
gh run list --branch main --status failure
gh run view
gh pr checks

別リポジトリを指定する

多くの gh コマンドは、カレントディレクトリのリポジトリを対象にします。別のリポジトリを操作したい場合は、継承オプションの -R / --repo で OWNER/REPO を指定できます。これは Issue、PR、Actions などのコマンドでよく使います。[6][9][12]

コード例（言語名: shell）:

gh issue list -R cli/cli
gh pr list -R cli/cli --state merged
gh run list -R owner/repo

JSON 出力を使う

一覧や確認結果をスクリプトで使うときは、対応コマンドの --json で出力フィールドを指定し、必要なら --jq で絞り込みます。gh repo view、gh issue list、gh pr list、gh run list などは、公式マニュアルで利用可能な JSON fields を列挙しています。[4][6][9][12]

コード例（言語名: shell）:

gh repo view --json nameWithOwner,visibility,url
gh pr list --json number,title,url --jq '.[] | "\(.number) \(.title)"'
gh issue list --json number,title,state
gh run list --json databaseId,status,conclusion,displayTitle

最初に覚える最小セット

日常用途なら、まずは次の 10 個を手癖にするとよいです。認証は gh auth login、状態確認は gh auth status、リポジトリ確認は gh repo view、clone は gh repo clone OWNER/REPO、Issue は gh issue list と gh issue view、PR は gh pr list と gh pr view、Actions は gh run list、別リポジトリ指定は -R OWNER/REPO です。作成系の gh issue create と gh pr create は、確認系に慣れてから使うのが安全です。[2][3][4][5][6][7][9][10][12]
