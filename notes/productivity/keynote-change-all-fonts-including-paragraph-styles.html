Keynote の全文字フォントと段落スタイルの既定を一括変更する方法

作成日: 2026-06-23

更新日: 2026-06-24

カテゴリ: productivity

元の質問

Keynote ファイルの中の文字のフォントを、図のタイトルやキャプションも含めて全部変えたい。さらに段落スタイルの既定フォントも変えて、新しく入力した文字も常にそのフォントになるようにするには、どのような戦略を取ればよいか？

回答

「見えている文字を変える」ことと「これから入力する文字の既定を変える」ことは、Keynote では別の問題です。前者は AppleScript で全オブジェクトのフォントを上書きし、後者は .key 内部のスタイル定義（IWA ファイル）を直接書き換える、という二段構えにすると確実でした。以下は実際に Mac の Keynote ファイルで全文字を Lucida Grande に変えたときの戦略です。

実行には KeyFontChanger を使える

この二段構えを macOS 用 CLI ツールとして実装した KeyFontChanger を公開しました。図のタイトル、キャプション、グループ内テキスト、表セルなどの表示中の文字と、段落スタイル・表スタイルの既定フォントをまとめて変更できます。変更前の自動バックアップと、変更結果を確認する検証コマンドも備えています。

リポジトリを取得し、uv で依存関係を準備します。

git clone https://github.com/kanonundgigue/KeyFontChanger.git
cd KeyFontChanger
uv sync
./scripts/install-command.sh

変更先には、フォントブックの表示名ではなく PostScript 名を指定します。基本操作は次の 3 コマンドです。

# 利用できる PostScript フォント名を確認
key-font-changer list-fonts --filter Hiragino

# 表示中の文字と内部スタイルを一括変更
key-font-changer change presentation.key HiraginoSans-W3

# 期待したフォントに統一されたか検証
key-font-changer verify presentation.key HiraginoSans-W3

change は通常、以下で説明する Stage 1 と Stage 2 を続けて実行します。Keynote でファイルを開いたまま処理する場合は --stage1-only を使い、IWA を直接編集する Stage 2 は Keynote を閉じてから --stage2-only で実行します。手順の背景や独自実装が必要な場合は、以下の仕組みを参照してください。

全体戦略（二段構え）

  1. 表示中の文字は AppleScript で全スライド・全オブジェクトを走査して上書きする。
  2. 段落スタイル・表スタイルの既定フォントは、AppleScript では触れないため、.key パッケージ内の Index/*.iwa を解析して書き換える。
  3. 各段階のあとに必ず再オープン検証を行い、保存後も設定が残っているか確認する。

事前準備：必ずバックアップを取る

どちらの工程も破壊的になり得ます。工程ごとに .key をコピーしておくと、失敗しても 1 手で戻せます。

cp -p deck.key deck_before_font_change.key

段階 1：フォント名は「PostScript 名」で指定する

最初の落とし穴がこれです。AppleScript で set font ... to "Lucida Grande" のように表示名（ファミリー名）を渡すと、エラーは出ないのに何も変わらないことがあります。Keynote は内部的に PostScript 名（例: LucidaGrande）を期待するためです。インストール済みフォントの PostScript 名は次で確認できます。

osascript -e 'use framework "AppKit"' \
  -e 'set fnts to (current application'"'"'s NSFontManager'"'"'s sharedFontManager()'"'"'s availableFonts()) as list' \
  -e 'return fnts'

「Lucida Grande」は LucidaGrande、ボールドは LucidaGrande-Bold という具合です。

段階 2：AppleScript で全オブジェクトを再帰的に上書きする

テキストは本文テキストボックスだけでなく、図形・表セル・グループの中・プレースホルダにも入っています。トップレベルだけを回すと、グループ内の図キャプションなどが取り残されます。そこで iWork items を再帰的にたどるのが要点です。

on setFont(obj, f)
  set n to 0
  tell application "Keynote"
    try
      set font of object text of obj to f
      set n to n + 1
    end try
    try
      repeat with c in cells of obj
        try
          set font of c to f
          set n to n + 1
        end try
      end repeat
    end try
    try
      repeat with child in iWork items of obj
        set n to n + (my setFont(child, f))
      end repeat
    end try
  end tell
  return n
end setFont

これをスライドの各 iWork item と、default title item / default body item に適用します。最後に save します。

段階 3：再オープンして「定着」を確認する

保存直後のメモリ上の値ではなく、閉じて開き直した後のフォントを読み返すのが大事です。これで「保存したつもりが戻っていた」を防げます。

段階 4：段落スタイルの既定フォントを変える（IWA 編集）

ここまでで「いま見えている文字」は変わりますが、段落スタイルの定義そのものには古いフォントが残ります。これを直さないと、新規入力やスタイル再適用で古いフォントが復活します。AppleScript からはスタイル定義を編集できないため、.key（実体は ZIP）の中の Index/*.iwa を解析します。

.iwa は Snappy 圧縮された Protocol Buffers で、フォント名は ParagraphStyleArchive.char_properties.font_name や TableStyleArchive.table_properties.master_font_family といったフィールドに入っています。Python の keynote-parser を使うと、解凍・再圧縮・長さ再計算まで面倒を見てくれます。

python3 -m venv /tmp/kv
/tmp/kv/bin/pip install keynote-parser protobuf python-snappy

これらのフィールドだけを新フォント名に置換し、書き戻します。バイト列を単純置換すると Protobuf の長さ表記や IWA のチャンク長が壊れるため、必ずメッセージ単位で書き換えて再シリアライズします。

最大の落とし穴：再 ZIP で画像が消える

ここが今回いちばんはまった点です。.key 全体を Python でまるごと作り直すと、スクリーンショット系画像の非 ASCII ファイル名や ZIP メタデータが壊れ、画像エントリが脱落して表示できなくなりました（109 個の Data/ が 61 個に減少）。

対策は単純で、画像（Data/）には一切触れず、変更した Index/*.iwa だけを既存 ZIP に上書きすることです。元ファイルを ZIP として開いたまま、対象エントリのみ差し替えます。

# 変更した iwa だけを既存の .key に無圧縮で上書きする
cd /tmp/work    # ここに Index/*.iwa を展開・編集済み
zip -0 -X -q -u "/abs/path/deck.key" Index/DocumentStylesheet.iwa Index/Slide-XXXX.iwa

差し替え後に Data/ のエントリ数が元と同じであることを必ず確認します。

最終検証

  • Keynote で開き直し、全テキストオブジェクトのフォントが目的のフォントになっているか走査する。
  • IWA を再解析し、古いフォント名の文字列が 0 件であることを確認する。
  • 画像エントリ数が変更前と一致しているか（脱落していないか）確認する。
  • PDF へ書き出し、ページ数と、スクリーンショットを含むページの見た目を確認する。

まとめ：戦略の型

通常の作業では、この一連の処理を実装した KeyFontChanger を使うのが簡単です。その内部では、「表示の上書き（AppleScript・再帰）」＋「既定の書き換え（IWA 編集・画像非破壊）」＋「再オープン／再解析による検証」という三点セットを採用しています。独自に処理する場合も、フォント名は PostScript 名で渡すこと、グループ内まで再帰すること、再 ZIP では画像を触らないことの 3 つが重要です。
