Homebrew と MacPorts の違いは？

作成日: 2026-07-03

カテゴリ: productivity

元の質問

普段はhomebrewを使っていますが、Macportを新規に導入することを検討しています。homebrewとの違いは？

回答

すでに Homebrew で困っていないなら、MacPorts は「置き換え」ではなく「必要なときだけ別系統として足す」くらいに考えるのが現実的です。Homebrew は日常的な CLI ツールや GUI アプリの導入が速く、MacPorts は `/opt/local` に独立した環境を作り、依存関係やビルド variant を MacPorts 側で管理しやすい、という性格の違いがあります。Homebrew の標準 prefix は Apple Silicon で `/opt/homebrew`、Intel Mac で `/usr/local` で、標準 prefix は bottle と呼ばれる事前ビルド済みバイナリの利用条件にも関係します。[1] MacPorts は標準で `/opt/local` に入れ、OS やベンダー提供ソフトと混ざらない private sandbox に閉じ込める設計を説明しています。[2]

結論だけ先に書くと、普段使いは Homebrew のままで十分です。MacPorts を入れる価値が出るのは、Homebrew の formula ではビルド条件を変えにくいものを variant 付きで入れたい、研究・開発用に Homebrew と別の依存関係ツリーを持ちたい、または MacPorts にある port のほうが目的に合う、という場合です。

大きな違い

観点 Homebrew MacPorts
標準の置き場所 Apple Silicon は `/opt/homebrew`、Intel Mac は `/usr/local`。[1] 標準は `/opt/local`。設定上の prefix も既定値は `/opt/local` とされています。[3]
権限モデル 通常の `brew install` では sudo を使わない思想で、単一ユーザー利用を主に想定します。[4] 多くの操作を `sudo port install ...` の形で実行します。公式 Guide の install 例も `sudo port install apache2 ...` です。[5]
バイナリかソースか 利用可能なら bottle を使うのが標準で、標準 prefix から外れるとソースビルドになりやすくなります。[1] prebuilt archive があれば使い、なければソースからビルドする運用です。buildfromsource の既定は必要時にビルドへ戻る `ifneeded` です。[6]
ビルドオプション 多くの利用では既定ビルドをそのまま使う前提です。独自変更は formula 編集など開発者寄りの作業になります。 port ごとに `+openldap` や `-preforkmpm` のような variant を選べる設計です。[7]
GUI アプリ Homebrew Cask が `.app`、`.pkg`、フォント、入力メソッドなどの artifact を扱えます。[8] native macOS application bundle の既定インストール先は `/Applications/MacPorts` です。[3]
古い版の扱い `brew upgrade` 後は古い formula を自動 cleanup する運用が基本です。[9] upgrade 後も古い inactive version を安全策として残し、必要なら `sudo port uninstall inactive` で掃除します。[10]

併用するときの考え方

Homebrew と MacPorts は標準の置き場所が違うため、同じ Mac に入れること自体は可能です。ただし、同じ名前のコマンドやライブラリが両方に入ると、どちらが使われるかは `PATH` やビルド時の探索順に依存します。実務上は、日常の shell では Homebrew を優先し、MacPorts が必要な作業だけ明示的に `/opt/local/bin` を前に出す、という運用が安全です。

特に Python、Ruby、Perl、OpenSSL、GDAL、Qt、TeX、Fortran 関連のように依存関係が大きいものを両方で入れる場合は、混ぜてビルドしないほうがよいです。たとえば Homebrew の Python から MacPorts のライブラリを拾ったり、その逆をすると、原因の追いにくいリンクエラーや実行時エラーになります。これは公式文書の個別警告というより、両者が別 prefix に依存関係ツリーを作ることから来る実務上の注意です。

MacPorts を足すとよい場面

  • 特定ソフトを variant 指定で入れたい。例: GUI 有無、LDAP 有無、universal build などを明示したい。
  • 研究・解析用に、Homebrew とは別の依存関係セットを固定したい。
  • Homebrew では目的の formula/cask がない、または MacPorts の port のほうが更新・依存関係・オプションの都合がよい。
  • 古い inactive version を一時的に残しながら、更新後の戻しやすさを重視したい。

Homebrew のままでよい場面

  • CLI ツールを手早く入れて更新したい。
  • GUI アプリも `brew install --cask ...` でまとめて管理したい。
  • ビルドオプションを細かく変える予定がない。
  • チームや記事・README の手順が Homebrew 前提になっている。

最初に入れるなら試すコマンド

MacPorts 導入後は、いきなり既存 Homebrew 環境と同じものを大量に入れず、MacPorts でだけ試したいものを 1 つに絞るのがよいです。MacPorts は macOS ではまず Command Line Tools、必要に応じて Xcode を使います。公式 Guide は、多くの port では full Xcode なしでもよいが、`xcode-select --install` で Command Line Tools を入れる手順を示しています。[11]

コマンド例（言語名: shell）:

# MacPorts 側の更新
sudo port selfupdate

# port を探す
port search gdal
port info gdal
port variants gdal

# variant を確認してから入れる
sudo port install gdal

# 古い inactive version の確認と掃除
port installed inactive
sudo port uninstall inactive

普段の Homebrew 環境を壊したくないだけなら、まずは MacPorts の PATH を常時有効にせず、必要なターミナルセッションだけで使うのが無難です。たとえば `.zshrc` にすぐ書き込む前に、試験用 shell で `export PATH="/opt/local/bin:/opt/local/sbin:$PATH"` を一時的に実行し、目的の port が問題なく動くか確認します。

まとめ

Homebrew は「Mac でソフトをすばやく入れる日常用の標準装備」に近く、MacPorts は「独立した prefix に、より ports 的な依存関係ツリーと variant を持つ別環境」です。普段 Homebrew を使っているなら、MacPorts を入れる理由は「Homebrew より優れているから」ではなく、「別 prefix・別依存関係・variant が必要だから」と考えると判断しやすいです。
