Docker の基本的な使い方は？

作成日: 2026-07-04

カテゴリ: productivity

元の質問

Dockerの基本的な使い方は？

回答

Docker は、アプリケーションと実行に必要なものを image としてまとめ、container という隔離された実行単位で動かすための仕組みです。Docker 公式は、Docker を「開発、配布、実行」のためのオープンプラットフォームとして説明し、container は image から作られる実行可能なインスタンスだと整理しています。[1] まずは「image を取る、container を起動する、ログを見る、止める、消す、自分の Dockerfile から image を作る」という流れを覚えるのが基本です。

最初に押さえる用語

  • image: container を作るための読み取り専用テンプレート。OS の最小環境、言語ランタイム、アプリ、設定などを含められる。
  • container: image から起動した実行中または停止中の実体。削除すると、永続化していない変更は消える。
  • Dockerfile: image の作り方を書いたテキストファイル。FROM、COPY、RUN、CMD などを並べる。
  • registry: image の置き場。Docker Hub は既定で使われる公開 registry として説明されています。[1]
  • volume / bind mount: container の外にデータを置く仕組み。volume は Docker 管理の永続領域、bind mount はホスト側の具体的なディレクトリを container に見せる方法です。[4][5]

既存 image を動かす

最初は自分で image を作らず、公開 image を一度だけ実行して感覚をつかむとよいです。Docker 公式の概要ページでも、docker run -i -t ubuntu /bin/bash のように Ubuntu image から対話的な container を起動する例が示されています。[1]

コード例（言語名: shell）:

# Ubuntu の shell に入る。ローカルになければ image が取得される
docker run --rm -it ubuntu:24.04 bash

# container の中
cat /etc/os-release
exit

--rm は container 終了時に自動削除する指定です。試しに動かすだけなら便利ですが、container の中で作ったファイルも残りません。残したいデータは volume や bind mount に逃がします。

Web サーバーをポート公開して動かす

container 内のアプリを Mac やブラウザから見るには、host 側の port と container 側の port を対応させます。Docker Docs は -p HOST:CONTAINER 形式で port mapping を作ると説明しており、例では container の 3000 番を host の 127.0.0.1:3000 に公開しています。[2]

コード例（言語名: shell）:

# nginx を localhost:8080 で見る
docker run --name web-demo -d -p 127.0.0.1:8080:80 nginx:alpine

# 状態確認
docker ps

# ログを見る
docker logs web-demo

# 停止して container を消す
docker stop web-demo
docker rm web-demo

-d は background 実行、--name は container 名、-p 127.0.0.1:8080:80 は host の localhost 8080 番を container の 80 番へ渡す指定です。開発中に外部公開する必要がないなら、host 側を 127.0.0.1 に絞ると意図しない LAN 公開を避けやすくなります。

自分の Dockerfile から image を作る

アプリを container 化する場合は、プロジェクト直下に拡張子なしの Dockerfile を置き、docker build -t 名前 . で image を作ります。Docker の workshop では、Dockerfile を使って image を作り、-t で人間が扱いやすい名前を付け、末尾の . で現在ディレクトリを build context にする流れが説明されています。[3]

コード例（言語名: shell）:

# Dockerfile の例
cat > Dockerfile.example <<'EOF'
FROM python:3.12-slim
WORKDIR /app
COPY . .
CMD ["python", "-m", "http.server", "8000"]
EOF

# 実際に使う場合は Dockerfile という名前にしてから build する
docker build -t my-python-server .
docker run --rm -p 127.0.0.1:8000:8000 my-python-server

実運用では、.env、秘密鍵、巨大な出力ディレクトリ、cache などを image に入れないように .dockerignore を置きます。Docker Compose Quickstart でも、build context に不要なファイルを送らないために .dockerignore を使う意義が説明されています。[6]

データを残す: volume と bind mount

container の書き込み層は container の寿命に強く結びつくため、container を削除しても残したいデータは別に扱います。Docker Docs は、container が削除されると通常の file changes も削除される一方で、volume は個別 container の lifecycle を越えてデータを残す storage mechanism だと説明しています。[4]

コード例（言語名: shell）:

# Docker 管理の volume にデータを残す
docker volume create app-data
docker run --rm -v app-data:/data ubuntu:24.04 bash -lc 'date >> /data/history.txt'
docker run --rm -v app-data:/data ubuntu:24.04 cat /data/history.txt

# ホストの現在ディレクトリを container に読み書きさせる
docker run --rm -it -v "$PWD":/work -w /work ubuntu:24.04 bash

開発コードを container から直接見せたい場合は bind mount が便利です。Docker Docs は、host 側の特定ファイルやディレクトリを container と共有したい場合に bind mount を使うと説明し、より明示的な制御には --mount 構文を推奨しています。[5] 読み取りだけで十分なら :ro を付けると、container からホスト側ファイルを書き換えにくくできます。

複数 container は Docker Compose でまとめる

Web アプリと Redis、アプリと PostgreSQL のように複数 container を一緒に扱うなら、compose.yaml に services、ports、environment、volumes などを書き、docker compose up でまとめて起動します。Docker Compose Quickstart は、Compose が複数 service、network、volume を 1 つの YAML 設定で扱いやすくし、docker compose up で構成ファイルの service を作成・起動すると説明しています。[6]

コード例（言語名: shell）:

# compose.yaml の最小例
cat > compose.yaml <<'EOF'
services:
  web:
    image: nginx:alpine
    ports:
      - "127.0.0.1:8080:80"
EOF

docker compose up -d
docker compose ps
docker compose logs -f
docker compose down

よく使うコマンド早見表

目的 コマンド 見方
image を取得する docker pull ubuntu:24.04 明示的に registry から取る。docker run 時にも必要なら自動取得される。
container を起動する docker run --rm -it ubuntu:24.04 bash 1 回限りの対話実行。
動いている container を見る docker ps 停止中も含めるなら docker ps -a。
container 内で追加コマンドを実行する docker exec -it 名前 bash 既に動いている container に入る。
ログを見る docker logs -f 名前 -f は追尾。
停止する docker stop 名前 停止しても container は残る。
削除する docker rm 名前 container を消す。image や volume は別管理。
image を作る docker build -t app . 現在ディレクトリを build context として Dockerfile を読む。
Compose で起動する docker compose up -d compose.yaml の service 群をまとめて起動。

最初の判断基準

  • 「一時的に Linux コマンドを試す」なら docker run --rm -it ubuntu:24.04 bash。
  • 「Web アプリをローカルで見る」なら -p 127.0.0.1:ホスト側:コンテナ側。
  • 「DB や生成データを残す」なら named volume。
  • 「ソースコードを container に読ませて開発する」なら bind mount。
  • 「複数 service を毎回同じ構成で立てる」なら Docker Compose。
  • 「他人に配る、CI で使う、本番へ近づける」なら Dockerfile と .dockerignore を整える。

まとめ

Docker の基本は、image を材料に container を起動し、必要なら port、environment、volume、bind mount を足して、作業が終わったら止める、という操作です。単発なら docker run、自分のアプリなら Dockerfile と docker build、複数 container なら docker compose up、データを残すなら volume、と分けて考えると見通しがよくなります。
