Docker の基本的な使い方は？

作成日: 2026-07-04

更新日: 2026-07-04

カテゴリ: productivity

元の質問

Dockerの基本的な使い方は？

回答

Docker は、今使っている Mac や PC の中に、別の小さな実行環境を作ってアプリやコマンドを動かす道具です。たとえば「このプロジェクトだけ Ubuntu で動かしたい」「自分の Mac に直接いろいろ入れずに Web サーバーを試したい」「他の人と同じ環境でプログラムを動かしたい」ときに使います。

公式には、Docker はアプリケーションを開発、配布、実行するためのオープンプラットフォームで、container は image から作られる実行可能なインスタンスだと説明されています。[1] 初心者はまず、「image は起動元、container は実際に動いているもの」と覚えれば十分です。

まず結論: 最初はこの流れだけでよい

  1. Docker Desktop を起動する。
  2. docker run ... で container を起動する。
  3. docker ps で動いているか見る。
  4. docker logs ... で出力を見る。
  5. docker stop ... と docker rm ... で止めて消す。

最初から Dockerfile、network、volume、Compose を全部理解しようとすると混乱します。まずは「他の人が用意した image を、自分の PC で container として起動する」だけを触るのがよいです。

その前に: Docker の本体が起動しているか確認する

docker run は、Docker の命令を受け取る daemon が起動していないと動きません。Mac では Docker Desktop を使っている場合と、Colima を使っている場合があります。Colima は macOS と Linux 向けの container runtime 環境で、Docker runtime では colima start のあとに docker run hello-world や docker ps を使う例が公式 README に示されています。[7]

次のようなエラーが出る場合は、Docker コマンドの書き方ではなく、daemon が起動していない可能性が高いです。

failed to connect to the docker API at unix:///Users/kanon/.colima/default/docker.sock
check if the path is correct and if the daemon is running

Colima を使っているなら、まず状態を確認して、止まっていれば起動します。

コード例（言語名: shell）:

# Docker CLI がどの接続先を見ているか確認する
docker context ls

# Colima の状態を見る
colima status

# "colima is not running" なら起動する
colima start

# 起動後にもう一度試す
docker run --rm -it ubuntu:24.04 bash

Docker Desktop を使っている場合は、Docker Desktop アプリを起動してからもう一度 docker run を試します。docker context ls で現在の context が colima なのか default なのかを見ると、自分が Docker Desktop と Colima のどちらへ接続しようとしているかを確認できます。

最初に押さえる用語

  • image: Ubuntu、nginx、Python などの起動元。まだ動いていない。
  • container: image から起動した実体。動いている場合も、停止して残っている場合もある。
  • Docker Hub: 公開 image の置き場。docker run ubuntu:24.04 のように指定すると、ローカルになければ取得される。
  • port: Web サーバーなどへアクセスする入口。container の中の port と、自分の PC 側の port をつなぐ必要がある。
  • volume / bind mount: container の外にデータを置く仕組み。データを残したいときに後で使う。[4][5]

最初の練習: Ubuntu に入って出る

最初は自分で image を作らず、公開 image を一度だけ実行して感覚をつかむとよいです。次のコマンドは、Ubuntu の container を起動して、その中の shell に入ります。Docker 公式の概要ページでも、Ubuntu image から対話的な container を起動する例が示されています。[1]

コード例（言語名: shell）:

# Ubuntu の shell に入る。ローカルになければ image が取得される
docker run --rm -it ubuntu:24.04 bash

# ここから先は container の中
cat /etc/os-release
exit

このコマンドは、次のように読めます。

部分 意味 初心者向けの見方
docker run container を起動する Docker で何かを動かす基本形。
--rm 終了後に container を自動削除する 練習用。作業結果は残らない。
-it 対話操作できるようにする shell に入るときはだいたい付ける。
ubuntu:24.04 使う image 名 Ubuntu 24.04 の環境を使う。
bash container 内で実行するコマンド Ubuntu の中で shell を開く。

exit すると container から出ます。--rm を付けているので、出た時点でこの container は消えます。

次の練習: Web サーバーをブラウザで見る

Docker で Web アプリを動かすときは、container の中の port を、自分の PC 側の port に接続します。Docker Docs は -p HOST:CONTAINER 形式で port mapping を作ると説明しています。[2]

コード例（言語名: shell）:

# nginx という Web サーバーを background で起動する
docker run --name web-demo -d -p 127.0.0.1:8080:80 nginx:alpine

# 状態確認
docker ps

# ログを見る
docker logs web-demo

# 停止して container を消す
docker stop web-demo
docker rm web-demo

起動したら、ブラウザで http://127.0.0.1:8080/ を開きます。ここで重要なのは、nginx は container の中では 80 番 port で動いているが、自分の PC からは 8080 番として見ている、という点です。

  • -d: 画面を占有せず、background で動かす。
  • --name web-demo: container に web-demo という名前を付ける。
  • -p 127.0.0.1:8080:80: PC 側の 127.0.0.1:8080 を container 側の 80 に接続する。

開発中に外部公開する必要がないなら、host 側を 127.0.0.1 に絞ると意図しない LAN 公開を避けやすくなります。

Dockerfile は「自分用 image の作り方」

ここまでは、Ubuntu や nginx など、既にある image を使いました。自分のアプリを同じ形で何度も動かしたい場合は、Dockerfile を作って自分用 image を作ります。Docker の workshop では、Dockerfile を使って image を作り、docker build -t 名前 . のように build する流れが説明されています。[3]

コード例（言語名: shell）:

# Dockerfile という名前のファイルを作る
cat > Dockerfile <<'EOF'
FROM python:3.12-slim
WORKDIR /app
COPY . .
CMD ["python", "-m", "http.server", "8000"]
EOF

# Dockerfile から image を作る
docker build -t my-python-server .

# 作った image から container を起動する
docker run --rm -p 127.0.0.1:8000:8000 my-python-server

末尾の . は「今いるディレクトリを build に使う」という意味です。実運用では、.env、秘密鍵、巨大な出力ディレクトリ、cache などを image に入れないように .dockerignore も置きます。Docker Compose Quickstart でも、build context に不要なファイルを送らないために .dockerignore を使う意義が説明されています。[6]

データは何もしないと残らない

初心者が混乱しやすい点は、container の中で作ったファイルが永続保存とは限らないことです。Docker Docs は、container が削除されると通常の file changes も削除される一方で、volume は個別 container の lifecycle を越えてデータを残す storage mechanism だと説明しています。[4]

まずはこのように考えると分かりやすいです。

  • 練習だけなら --rm を付けて、container を使い捨てにする。
  • データベースの中身などを残したいなら volume を使う。
  • 自分の手元のプロジェクトフォルダを container から見せたいなら bind mount を使う。

コード例（言語名: shell）:

# Docker 管理の volume にデータを残す
docker volume create app-data
docker run --rm -v app-data:/data ubuntu:24.04 bash -lc 'date >> /data/history.txt'
docker run --rm -v app-data:/data ubuntu:24.04 cat /data/history.txt

# ホストの現在ディレクトリを container に読み書きさせる
docker run --rm -it -v "$PWD":/work -w /work ubuntu:24.04 bash

bind mount は、host 側の特定ファイルやディレクトリを container と共有する仕組みです。Docker Docs は、より明示的な制御には --mount 構文を推奨しています。[5] ただし最初は、-v "$PWD":/work が「今いるフォルダを container の /work として見せる」という意味だと分かれば十分です。

Docker Compose は「複数 container の起動メモ」

Web アプリだけなら docker run でも動かせます。しかし、Web アプリ、データベース、Redis のように複数 container が必要になると、毎回長いコマンドを打つのがつらくなります。そのときに compose.yaml へ設定を書き、docker compose up でまとめて起動します。Docker Compose Quickstart は、Compose が複数 service、network、volume を 1 つの YAML 設定で扱いやすくし、docker compose up で構成ファイルの service を作成・起動すると説明しています。[6]

コード例（言語名: shell）:

# compose.yaml という名前のファイルを作る
cat > compose.yaml <<'EOF'
services:
  web:
    image: nginx:alpine
    ports:
      - "127.0.0.1:8080:80"
EOF

# compose.yaml に書いた service を起動する
docker compose up -d
docker compose ps
docker compose logs -f
docker compose down

初心者向けコマンド早見表

目的 コマンド まずこう理解する
Ubuntu を試す docker run --rm -it ubuntu:24.04 bash Linux 環境に一時的に入る。
Web サーバーを試す docker run --name web-demo -d -p 127.0.0.1:8080:80 nginx:alpine ブラウザで 127.0.0.1:8080 を開く。
動いている container を見る docker ps 今 Docker で動いているものの一覧。
停止中も含めて見る docker ps -a 過去に作った container も見る。
ログを見る docker logs -f 名前 アプリの出力を見る。
停止する docker stop 名前 container を止める。まだ削除はしない。
削除する docker rm 名前 container を消す。image や volume は別。
image を作る docker build -t app . Dockerfile から自分用 image を作る。
Compose で起動する docker compose up -d compose.yaml に書いた複数 service を起動する。

初心者がよく混乱する点

  • Docker を入れただけでは何も動かない。 Docker Desktop を起動し、さらに docker run で container を起動します。
  • image と container は別物。 image は起動元、container は実際に起動したものです。
  • localhost で見えないときは port 指定を見る。 Web サーバーなら -p 127.0.0.1:8080:80 のような指定が必要です。
  • --rm を付けた container の中身は終了後に消える。 残す必要があるものは volume か bind mount を使います。
  • 最初から Dockerfile と Compose を書かなくてよい。 まずは既存 image を docker run で動かすところからで十分です。

まとめ

Docker の最初の理解は、「image から container を起動して、その中でアプリやコマンドを動かす」で十分です。最初に覚えるのは docker run、docker ps、docker logs、docker stop、docker rm です。Web アプリを見るときは port、データを残すときは volume や bind mount、毎回同じ環境を作りたいときは Dockerfile、複数 container をまとめたいときは Docker Compose、という順番で広げると混乱しにくくなります。
