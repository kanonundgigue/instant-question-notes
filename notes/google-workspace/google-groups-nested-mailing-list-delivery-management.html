Google グループを入れ子にしたメーリングリストの配信管理

作成日: 2026-06-12

カテゴリ: google-workspace

元の質問

Google groupのメーリングリストの中にGoogle groupのメーリングリストを登録した場合の、配信状況の管理方法は？

回答

Google Workspace の Google グループでは、あるグループを別のグループのメンバーとして追加できます。Google の用語では、追加先が親グループ、追加される側が子グループ、またはネストされたグループです。配信管理は「親グループに子グループを直接メンバーとして登録する」「子グループ側で実際のユーザーを管理する」「個別メッセージの到達確認は Email Log Search で追う」と分けて考えるのが基本です。

管理単位は親グループの直接メンバーと子グループのメンバーに分かれる

親グループに登録されるのは、子グループのメールアドレスです。子グループに所属する各ユーザーは、親グループの直接メンバーにはなりません。そのため、子グループのユーザーは親グループから自分で退会したり、親グループ上の自分のメンバー設定を変更したりできません。

運用上は、親グループでは「どの子グループを含めるか」を管理し、子グループでは「誰を含めるか」を管理します。複数の親グループに同じ子グループを入れている場合、子グループのメンバーを更新すれば、それらの親グループに間接的に反映されます。

子グループのメンバーが親グループの投稿を受け取る条件

Google の説明では、子グループのメンバーが親グループに投稿されたメッセージを受け取るには、子グループ側の「投稿できるユーザー」設定が関係します。子グループの Who can post を「組織内のすべてのユーザー」以上、たとえば「ウェブ上のすべてのユーザー」まで広げる必要がある場合があります。これが狭いと、親グループから子グループへの投稿として扱われたメッセージが子グループで受け付けられず、子グループのメンバーに届かないことがあります。

つまり、親グループに子グループを追加しただけでは十分ではありません。親グループ側のメンバー追加、子グループ側の投稿許可、子グループ側のメンバー配信設定をあわせて確認します。

管理画面で確認する場所

Google 管理コンソールでは、Directory の Groups から親グループを開き、メンバー一覧で直接メンバーを確認します。対応エディションでは、Direct and indirect members を開くことで、親グループの直接メンバーと、子グループ経由の間接メンバーを同じ一覧で確認できます。

ただし、間接メンバーのロールや禁止状態など、詳細なメンバー属性を確認したい場合は、実際にそのユーザーが所属している子グループを Google Groups 側で開いて確認します。

API で棚卸しする場合

Admin SDK Directory API の Member リソースでは、メンバーの種類として USER と GROUP が区別されます。子グループは親グループのメンバーとして GROUP 型になります。メンバー単位のメール配信設定は delivery_settings で表され、ALL_MAIL、DAILY、DIGEST、NONE、DISABLED などの値があります。

API や管理スクリプトで棚卸しする場合は、親グループのメンバー一覧から GROUP 型の子グループを抽出し、その子グループをさらに展開して、最終的な受信候補ユーザーの一覧を作るのが現実的です。循環した入れ子は API 上もエラーになるため、双方向に入れ子にする構成は避けます。

実際に届いたかは Email Log Search で見る

「このメッセージが誰に届いたか」「なぜ届かなかったか」を確認する用途では、Google 管理コンソールの Email Log Search を使います。Email Log Search は、組織内で送受信されたメールの詳細や配送ステップを確認でき、Google グループや組織内メーリングリスト宛のメッセージも検索対象にできます。

注意点として、Email Log Search でグループアドレスを受信者として検索した結果は、個々のグループメンバーへの配送情報を含まない場合があります。個別の到達状況まで追う場合は、メッセージ ID と特定の受信者アドレスで検索条件を絞ります。大きなグループではログ件数が多くなるため、対象メッセージと対象ユーザーを絞って調べる運用にします。

運用ルールの例

  • 親グループは配信先グループの束ね役として扱い、個人ユーザーをできるだけ直接入れない。
  • 子グループはチームや部門など、実際にメンバーを管理する単位にする。
  • 親グループに投稿したメールを子グループへ流したい場合は、子グループ側の投稿許可を確認する。
  • 配信対象者の棚卸しでは、親グループの直接メンバーと、子グループ経由の間接メンバーを分けて見る。
  • 個別の不達調査では、Email Log Search でメッセージ ID と受信者を指定して確認する。
  • 入れ子が深くなると説明責任と不達調査が難しくなるため、親子関係は浅く保つ。
