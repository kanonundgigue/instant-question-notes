このプロジェクトの使い方

作成日: 2026-06-12

更新日: 2026-06-13

カテゴリ: project

元の質問

このプロジェクトの使い方は？

回答

このプロジェクトは、疑問や調べたことを 1 テーマ 1 ページの静的 HTML 記事として残すためのノートです。普段の使い方は、質問を Codex や Claude Code に渡し、エージェントが記事ファイルとトップページの記事カードを更新して、人間が差分を確認してからコミットや公開を指示する流れです。

基本の流れ

  1. 疑問や調べたいことを Codex / Claude Code に渡します。
  2. エージェントが必要に応じて一次情報を確認し、notes/<category>/<slug>.html に記事を作ります。
  3. エージェントが index.html の記事一覧にカードを追加します。
  4. エージェントが成果物を人間が確認できる形にします。
  5. ローカル運用では、エージェントは記事と index.html を更新して差分を提示し、コミットせずに止まります。記事に追加修正を入れたいときは、続けて指示できます。
  6. コミットしてよい状態になったら、commitして と指示します。エージェントはそこでコミットし、push はしません。
  7. 公開してよい状態になったら、pushして または 公開して と明示します。エージェントは main ブランチへ直接 push し、GitHub Pages のデプロイを動かします。
  8. 作業ブランチや PR は、ユーザーが明示的に求めた場合だけ使います。

質問だけ送れば記事化される

このリポジトリでは、ユーザーが疑問や調べたいことだけを送った場合、特に断りがなければ記事作成依頼として扱います。毎回「記事にして」と書く必要はありません。このプロジェクト自身の使い方や運用ルールに関する質問も記事化対象です。

会話だけで済ませたい場合は、「説明だけ」「記事にしないで」「相談だけ」のように明示します。

質問例（言語名: text）:

uv.lock はいつ更新される？
GitHub Actions の permissions は何を指定すべき？
Python の pathlib.Path と os.path はどう使い分ける？
引っ越し前に住所変更が必要なサービスは？
このプロジェクトの使い方は？

記事の置き場所と書式

記事は notes/_template/article.html を元に作り、カテゴリ名と記事ファイル名は英小文字のケバブケースにします。記事ページには、タイトル、作成日、後日修正時の更新日、カテゴリ、元の質問、構造化された本文、参照 URL、トップページへの戻りリンクを置きます。

既存記事を後日修正し、本文、参照 URL、運用方針、結論など読者の理解に影響する内容を変えた場合は、記事ページのメタ情報と index.html の記事カードに更新日を追加または更新します。作成日は初回作成日として残します。記事一覧は、更新日がある記事は更新日、更新日がない記事は作成日を基準にした日付降順で並べます。

記事固有の CSS は原則使わず、記事ページからは ../../styles/main.css を読み込みます。ビルドツールや静的サイトジェネレータは使いません。

ローカルで読む方法

ビルド工程はありません。ローカルでは index.html をブラウザで開けば記事一覧を確認できます。記事ページはトップページのカードから移動します。

公開の仕組み

main ブランチへの push、または手動実行で GitHub Pages のワークフローが動きます。ワークフローは公開対象だけを _site に集め、index.html、styles/、images/、notes/ をデプロイします。ただし notes/_template/ は除外されます。

AGENTS.md、CLAUDE.md、README.md、plan.md、cleanup.sh などの運用ドキュメントは公開対象ではありません。

運用ルールの正本

エージェント向けの正本は AGENTS.md です。CLAUDE.md は Claude Code 向けの補足で、現在は AGENTS.md に従うよう案内しています。運用に迷った場合は、まず AGENTS.md を確認します。

注意点

  • 記事は日本語で書きます。
  • 推測を書く場合は「推測」と明示します。
  • 機密情報、個人情報、未公開の認証情報は記事に含めません。
  • コミット、push、公開は人間が明示した場合だけ実行します。
  • 作業ブランチや PR は、ユーザーが明示的に求めた場合だけ使います。
  • ファイル削除が必要な場合は直接削除せず、cleanup.sh に削除コマンドを追記して確認を待ちます。
