Claude や Codex のスラッシュコマンドを活用する方法

作成日: 2026-07-07

カテゴリ: ai-coding-agents

元の質問

claudeやcodexのスラッシュコマンドをうまく活用できていない。活用方法は？

回答

スラッシュコマンドは「機能一覧を覚えるもの」ではなく、作業の局面を素早く切り替えるための入口として使います。Claude Code なら /plan、/compact、/code-review、/debug のような組み込みコマンドや bundled skill を使い、繰り返す手順は Skills として /skill-name で呼び出すのが現在の中心です。Claude Code では custom commands が Skills に統合され、既存の .claude/commands/ も動きますが、新しく作るなら Skills を優先します。[1][2]

Codex では、CLI なら /model、/permissions、/plan、/review、/status などでセッションを制御できます。Codex app や IDE 拡張にも /plan、/goal、/review、/status などがありますが、使えるコマンドは環境とアクセス権で変わります。[3][4][5]

まず覚える分類

分類 使う場面 例
組み込みコマンド 今のセッションの状態、権限、モデル、レビュー、文脈量を操作する /status、/plan、/permissions、/compact、/review
Skills 毎回同じ手順で調査、レビュー、実装、検証、記事化を行う /summarize-changes、/deploy-staging、/write-note
旧カスタムプロンプト Codex で手元の ~/.codex/prompts にある古い再利用プロンプトを呼ぶ /prompts:draftpr

Codex の custom prompts は deprecated で、再利用したい指示は Skills を使うよう案内されています。custom prompts はローカルの ~/.codex に置く明示呼び出し用プロンプトなので、リポジトリで共有したい手順や、Codex に自動選択させたい手順には向きません。[6]

活用の順序

  1. まず、いまの作業局面を切り替える組み込みコマンドだけを使います。大きい変更前は /plan、差分確認は /diff や /review、長い会話では /compact、権限確認は /status や /permissions です。
  2. 次に、自分が毎回貼っている指示文を 3 つだけ拾います。たとえば「差分を要約してリスクを出す」「PR レビューコメントを直す」「このリポジトリの公開記事を作る」です。
  3. その指示文を Skills にします。Claude Code なら .claude/skills/<name>/SKILL.md、Codex なら .agents/skills/<name>/SKILL.md やユーザー配下の skills に置く、という考え方です。Codex の Skills は CLI、IDE 拡張、app で利用でき、タスク固有の指示、参照、スクリプトをまとめられます。[7]
  4. 最後に、よく使うものだけ明示呼び出しします。Claude Code では /skill-name、Codex では $skill-name や slash menu からの選択を使い、説明文に合う作業は暗黙起動に任せます。

最初に作ると効果が出やすいコマンド

名前 中身 向いている理由
summarize-changes 未コミット差分を読み、変更点、リスク、検証不足、commit message 案を出す 毎回の作業終盤で使え、失敗しても破壊的でない
fix-review-comments PR コメントやレビュー指摘を読み、実装するもの、保留するもの、質問するものに分ける 判断基準を固定しやすく、指摘の取りこぼしを減らせる
write-note 質問を記事化し、一次情報、引用、トップページ導線、検証まで進める このリポジトリのように手順が決まっている作業ほど効果が大きい
run-and-verify アプリを起動し、画面確認、テスト、lint、失敗ログの要約まで行う 「実行して確認する」を習慣化できる

Claude Code での考え方

Claude Code では、セッション制御は組み込みコマンド、繰り返し作業は Skills に寄せます。公式ドキュメントは、同じ指示、チェックリスト、複数手順を何度も貼っているなら Skill を作る場面だと説明しています。また、Skill の本文は使われたときだけ読み込まれるため、長い手順や参照を CLAUDE.md に常駐させるより文脈を節約できます。[1]

実務では、CLAUDE.md には恒久ルールだけを置き、作業手順は Skills に逃がします。たとえば「PR をレビューして修正まで行う」「デバッグログを読んで再現手順を作る」「発表資料の章立てを直す」のように、成果物と検証が決まっているものを /review-pr、/debug-failure、/revise-slides として切り出します。

Codex での考え方

Codex は surface によって slash command の粒度が違います。CLI はモデル、権限、IDE 文脈、差分、メモリ、プラグイン、フックなどを細かく操作できます。app と IDE 拡張は、/plan、/goal、/review、/status のように、スレッドやレビューの進行を制御するものが中心です。[3][4][5]

Codex で再利用ワークフローを作るなら、旧 custom prompts ではなく Skills を基本にします。Skill は SKILL.md、任意の scripts/、references/、assets/ を持てるため、単なる長いプロンプトより「手順 + 参照 + 検証」をまとめやすいです。[7]

小さな Skill の例

最初は、ファイル編集をしない read-only 系の Skill から始めると失敗が少ないです。以下は「差分の要約とリスク確認」を作るときのシェル例です。

コマンド例（言語名: shell）:

mkdir -p .agents/skills/summarize-changes
$EDITOR .agents/skills/summarize-changes/SKILL.md

SKILL.md には、次の要素だけを短く入れます。

  • name: コマンド名または表示名。
  • description: いつ使うか。暗黙起動の精度に効きます。
  • 手順: 何を読み、何を出すか。
  • 出力形式: 変更点、リスク、検証不足、次のアクション。

増やしすぎないルール

スラッシュコマンドは数が増えるほど使われなくなります。最初は「開始」「調査」「実装」「レビュー」「終了」の 5 局面に 1 つずつで十分です。

  • 開始: /plan または計画 Skill。
  • 調査: 公式情報確認やコード探索の Skill。
  • 実装: 対象ファイル、禁止事項、完了条件を固定した Skill。
  • レビュー: 差分、テスト、セキュリティ、不要変更を見る Skill。
  • 終了: 検証結果、残リスク、commit message 案を出す Skill。

うまく使えていないと感じる場合は、まず slash menu を眺めるより、直近 10 回の依頼文から「同じ文を貼っているところ」を探す方が早いです。その繰り返し文を 1 つだけ Skill 化し、1 週間使ってから次を増やすと定着します。
