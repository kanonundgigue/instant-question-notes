Claude Code Fable 5 で設計を統括し、Codex に実装させる方法

作成日: 2026-06-13

カテゴリ: ai-coding-agents

元の質問

claude code Fable5でオーケストレーションして実装をcodexでする方法を具体的に教えて

回答

実用上は、Claude Code Fable 5 を「リードエージェント」、Codex を「実装エージェント」として分けます。Claude Code は要求の分解、設計判断、タスク順序、レビュー観点の整理を担当し、Codex はリポジトリ内で実際にファイルを編集し、テストや lint を回して差分を作ります。

最重要のルールは 1 つだけです。同じファイルを Claude Code と Codex の両方に編集させないこと。最初は「Claude が計画を作る → 人間が承認する → Codex が実装する → Claude が差分をレビューする」という直列ループにするのが安全で、慣れてから並列化を検討します。

前提

Anthropic のモデル一覧では、Claude Fable 5 はモデル ID claude-fable-5 で提供される最上位モデル（Opus より上位の新ティア）で、1M トークンのコンテキストを持ち、長い自律作業や高度な推論向けと説明されています。Claude Code の subagent 設定でも model: fable を指定できます。設計判断とレビューという「考える側」をここに置く理由です。

Codex 側は、リポジトリの AGENTS.md を自動で読み、sandbox（read-only / workspace-write / danger-full-access）と approval ポリシーで権限を制御しながらファイル編集とテスト実行を進められるため、「手を動かす側」に向きます。

おすすめの役割分担

役割 担当 やらせないこと
Claude Code Fable 5 要件整理、設計分解、タスク発行、Codex の出力レビュー、次の指示作成 Codex が編集中のファイルを横から編集しない
Codex コード編集、テスト追加、テスト実行、lint、差分説明 要求範囲を超えた設計変更や大規模リファクタを勝手に広げない
人間 計画承認、権限承認、最終差分確認、commit / push 判断 承認前に大きい自動実装を走らせない

最小フロー

  1. Claude Code を Fable 5 または Fable 相当のモデルで開きます。
  2. Claude Code に「実装は Codex に渡す。あなたは計画とレビューだけを担当」と明示します。
  3. Claude Code に実装タスクを 1 つずつ codex exec 用プロンプトへ変換させます。
  4. Codex を workspace-write sandbox で実行し、必要な編集と検証だけをさせます。
  5. Codex の差分を Claude Code にレビューさせます。
  6. 問題があれば Claude Code が修正指示を作り、もう一度 Codex に渡します。

Claude Code への最初のプロンプト

まず Claude Code 側に、オーケストレーターとしての境界を固定します。

プロンプト例（言語名: text）:

あなたは Claude Code Fable 5 のリードエージェントです。
このリポジトリでは、あなたは設計、分解、レビュー、Codex への実装指示作成だけを担当してください。

実装のファイル編集は Codex に任せます。
あなた自身は、Codex が編集する対象ファイルを直接編集しないでください。

進め方:
1. まずリポジトリのルールと関連ファイルを読んでください。
2. 実装計画を小さなタスクに分割してください。
3. 各タスクについて、Codex に渡す `codex exec` 用の具体的なプロンプトを作ってください。
4. Codex 実行後は、差分、テスト結果、残リスクをレビューしてください。
5. 追加修正が必要なら、次の Codex 用プロンプトを作ってください。

Codex に渡す実装プロンプトの型

Codex には「目的、対象ファイル、禁止事項、完了条件」を明確に渡します。Claude Code には、この型で毎回プロンプトを生成させると安定します。

プロンプト例（言語名: text）:

目的:
<実装したい変更を 1 つだけ書く>

対象:
- <主に触ってよいファイルやディレクトリ>

制約:
- リポジトリの AGENTS.md に従う
- 関係ないリファクタはしない
- 既存の公開 API や UI 文言を変える場合は理由を説明する
- テストを追加または更新できる場合は行う

完了条件:
- 変更内容を実装する
- 関連するテスト / lint / 型チェックを実行する
- 実行できなかった検証があれば理由を明記する
- 最後に変更ファイル、検証結果、残リスクを短くまとめる

Claude Code から Codex CLI を呼ぶ例

Claude Code がシェル実行できる環境なら、Claude Code に次のようなコマンドを実行させます。codex exec は非対話実行のため、途中で人間の承認を待つ運用には向きません。sandbox を workspace-write に固定し、サンドボックスの境界内で完結するタスクだけを渡します。danger-full-access は自動化フローでは渡しません。

コマンド例（言語名: shell）:

codex exec --sandbox workspace-write "目的:
ログインフォームのバリデーションエラー表示を改善する。

対象:
- src/features/auth/
- src/components/form/

制約:
- AGENTS.md に従う
- 関係ないリファクタはしない
- 既存テストがあれば更新する

完了条件:
- 実装する
- 関連テストを実行する
- 変更内容と検証結果を報告する"

Codex 実行後、Claude Code には次を渡します。

プロンプト例（言語名: text）:

Codex が実装しました。
git diff とテスト結果を読んで、以下をレビューしてください。

- 要件を満たしているか
- 余計な変更がないか
- テスト不足がないか
- セキュリティやデータ損失のリスクがないか
- Codex に追加修正を依頼すべき具体的なプロンプト

subagent を使う場合

Claude Code 側に「Codex 実装指示を作る専門 subagent」を置くと、リードの会話を汚さずに実装依頼文を作れます。これは Codex を直接内包するものではなく、Codex に渡す指示を作るための Claude Code subagent です。

設定例（言語名: markdown）:

---
name: codex-task-writer
description: Codex に実装を渡すための、範囲の狭い実装プロンプトを作る
tools: Read, Glob, Grep
model: fable
---
あなたは Codex 実装タスクの設計者です。
リポジトリのルール、対象ファイル、完了条件を読み、Codex に渡す 1 タスク分のプロンプトだけを作ってください。

必ず含める項目:
- 目的
- 対象
- 制約
- 完了条件
- 実行すべき検証

禁止:
- あなた自身でファイルを編集しない
- 複数の独立した実装を 1 つのタスクに詰め込まない
- 未確認の前提を断定しない

このファイルは、プロジェクト単位なら .claude/agents/codex-task-writer.md、個人共通なら ~/.claude/agents/codex-task-writer.md に置きます。ファイルを直接編集して変更が反映されない場合は、Claude Code のセッションを再起動します。

並列化したい場合

Claude Code の agent teams は、リードと複数の teammate がタスク一覧とメッセージで協調する仕組みです。調査、レビュー、独立した新規モジュールには向きますが、同じファイルを複数エージェントが触る実装には向きません。

並列化するなら、Claude Code では「調査担当」「テスト観点担当」「レビュー担当」を並列にし、Codex の実装は 1 つの作業単位ずつ直列に流すのが安全です。どうしても複数 Codex 実装を並列にする場合は、Git worktree や別ブランチでファイル競合を分離します。

失敗しやすいパターン

  • Claude Code と Codex の両方に同じ実装ファイルを編集させる。
  • 「全部いい感じに実装して」と渡し、Codex の変更範囲が広がる。
  • Claude Code の計画承認なしに、長時間の自動実装を始める。
  • テスト実行や diff レビューを Codex の自己申告だけで終える。
  • 権限を広げすぎ、不要なネットワークアクセスやリポジトリ外書き込みを許す。

この形にするときの実務ルール

  • 共通ルールは AGENTS.md に置き、Codex が必ず読む状態にする。
  • Claude Code 用には CLAUDE.md を置き、AGENTS.md を正本として参照させる。
  • Claude Code は計画とレビュー、Codex は編集と検証、という境界を毎回プロンプトに入れる。
  • Codex には 1 回 1 タスクで渡す。大きい機能は「調査」「最小実装」「テスト追加」「仕上げ」に分ける。
  • 最後は Claude Code ではなく、人間が diff と検証結果を見て commit / push を判断する。
