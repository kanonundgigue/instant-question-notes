# instant-question-notes 仕様書

技術的な疑問を投げかけると、AI エージェント（Claude Code / Codex）が静的 HTML の記事として自動生成・蓄積する「技術質問ノート」システムの仕様書。

参考: [メモる速度で記事化する技術ノート（Zenn）](https://zenn.dev/ttaniguchi/articles/instant-question-html-notes)

---

## 1. コンセプト

- **「メモる速度で記事化する」**: 疑問が浮かんだ瞬間に質問を投げると、調査済みの記事が自動で生成され、知識が資産化される
- 調べ直しの手間を削減し、移動中・就寝前などの隙間時間を活用する
- 記事は静的 HTML として GitHub Pages で公開し、ブラウザからいつでも閲覧できる

## 2. システム概要

```
[質問者]
   │  ① 質問を投げる（音声入力 / テキスト）
   ▼
[AI エージェント（Claude Code / Codex、自宅 PC 上で稼働）]
   │  ② Web 調査・記事 HTML 生成
   │  ③ index.html（記事一覧）を更新
   ▼
[Git リポジトリ]
   │  ④ エージェントがコミット → 人間が目視確認 → 承認後にプッシュ
   ▼
[GitHub Pages]
      ⑤ 公開された記事をブラウザで閲覧
```

### 構成要素

| 要素 | 内容 |
|---|---|
| フロントエンド | 静的 HTML / CSS / SVG（ビルド不要、フレームワーク不使用） |
| ホスティング | GitHub Pages（公開リポジトリ） |
| バージョン管理 | Git（エージェントがコミットし、push は人間の承認後に行う） |
| 記事生成 | Claude Code または Codex への指示で自動生成 |
| 入力経路 | スマホの AI アプリ経由の遠隔指示、または PC 上で直接指示 |

## 3. リポジトリ構成

```
instant-question-notes/
├── index.html              # トップページ（全記事の一覧）
├── styles/
│   └── main.css            # 共通スタイルシート
├── images/                 # 図版（SVG 推奨）
├── notes/                  # 記事本体（カテゴリ別ディレクトリ）
│   ├── <category>/
│   │   └── <slug>.html     # 1ページ1テーマ
│   └── ...
├── .github/
│   └── workflows/
│       └── pages.yml       # GitHub Pages デプロイ（必要な場合）
├── AGENTS.md               # エージェント向け運用ルール（Codex 互換）
├── CLAUDE.md               # Claude Code 向け運用ルール（AGENTS.md を参照）
├── README.md               # プロジェクト説明（日本語）
├── cleanup.sh              # 削除コマンドの蓄積用（直接 rm しない）
└── .gitignore              # 機密情報・Git 管理外ファイルの除外
```

### 命名規則

- カテゴリ名: 英小文字ケバブケース（例: `python`, `git`, `climate-data`）
- 記事ファイル名: 内容を表す英小文字ケバブケースのスラッグ（例: `uv-lockfile-basics.html`）
- 1 ファイル = 1 テーマ。長くなる場合は分割して相互リンクする

## 4. デザイン仕様（案C: ペーパーノート）

紙のノートを思わせる、温かみのある落ち着いた配色。長文の技術記事を読み疲れせずに読めることを重視する。

### 4.1 デザインコンセプト

- **ペーパーノート**: クリーム色の紙地に、カードを並べて記事を整理する「手帳・ノート」のメタファー
- 隙間時間にスマホで読み返すことが主用途のため、目に優しく・落ち着いた配色とする
- 見出しはセリフ体で「読み物」らしさを出し、本文・コードはサンセリフ／等幅で可読性を確保する

### 4.2 カラーパレット

| 用途 | 色 | 備考 |
|---|---|---|
| 背景（紙地） | `#FAF6EE` | クリーム色のベース |
| カード背景 | `#FFFFFF` | 記事カード・本文ブロック |
| 境界線 | `#E6DDCC` | カード枠・区切り線 |
| 本文テキスト | `#3B3428` | 濃い茶系（黒は使わない） |
| サブテキスト（日付など） | `#9A8F7A` | 薄い茶系 |
| アクセント（カテゴリ・リンク） | `#D85A30` | コーラル系。見出し下線・リンク・カテゴリラベル |
| コードブロック背景 | `#3B3428` | 濃茶地に `#F5C4B3`（淡コーラル）の文字 |

ライト基調を正とし、ダークモード対応は将来拡張（9 章）とする。

### 4.3 タイポグラフィ

- **記事タイトル・サイト名**: セリフ体（`Noto Serif JP` など日本語セリフ）／`font-weight: 500`
- **本文**: サンセリフ（`Noto Sans JP` など）／`line-height: 1.8` で行間広め
- **コード**: 等幅フォント（`font-mono`）
- 見出しは Title Case や全大文字を使わず、素直な文章ケースで書く
- フォントは Web フォント（Google Fonts 等）を CDN から読み込むか、システムフォントにフォールバックする

### 4.4 レイアウト

- **トップページ（index.html）**
  - サイト名（セリフ体）＋ サブコピー「疑問をメモる速度で記事化する」をヘッダーに置く
  - 記事は**カードグリッド**（レスポンシブ 2 列、スマホでは 1 列）で並べる
  - 各カード: カテゴリラベル（コーラル）＋ 記事タイトル＋ 作成日
  - カードはカテゴリ横断で日付降順に並べる（カテゴリ見出しでのグルーピングは任意）
- **記事ページ**
  - 上部に「← 一覧に戻る」リンク（コーラル）
  - タイトル（セリフ体）＋ メタ情報（作成日・カテゴリ）
  - 本文は白カードまたは紙地に直接、1 カラムで配置（読みやすい行長 ≒ 全角 38〜42 文字で折り返す）
  - コードブロックは濃茶地に淡コーラル文字
  - 末尾に区切り線＋「参照」セクション（必須）

### 4.5 装飾の方針

- グラデーション・影・光彩は使わず、フラットな面と細い境界線で構成する
- カードの角丸は 6px 程度。過度な装飾はしない
- アイコンが必要な場合のみ最小限の SVG を `images/` に置く

## 5. 記事ページの仕様

### 5.1 必須構成要素

各記事 HTML は以下を含む:

1. **タイトル**（`<h1>`）: 質問の主旨を表す見出し
2. **メタ情報**: 作成日・カテゴリ・元の質問文
3. **本文**: 質問への回答。見出し（`<h2>`/`<h3>`）で構造化する
4. **コード例**: `<pre><code>` で記載。言語名を明記する
5. **参照 URL**: 記事末尾に**必ず**一次情報へのリンクを列挙する（信頼性確保のための必須要件）
6. **ナビゲーション**: トップページ（`index.html`）への戻りリンク

### 5.2 HTML テンプレート方針

- ビルド工程を持たないため、テンプレートは「雛形 HTML をエージェントが複製・編集する」方式とする
- 共通 CSS（`styles/main.css`）を読み込み、記事ごとの独自スタイルは原則持たない
- 図が必要な場合は SVG をインラインまたは `images/` 配下に置いて参照する
- レスポンシブ対応（スマホ閲覧が主用途のため、モバイルファーストで設計）

### 5.3 index.html（記事一覧）

- 全記事をカードグリッドで列挙する（カテゴリ見出しでのグルーピングは任意）
- 各エントリ: カテゴリラベル + 記事タイトル + 作成日 + リンク
- 新しい記事が上に来るよう日付降順で並べる
- 記事追加時にエージェントが自動で index.html を更新する

## 6. 運用フロー

主たる入力経路は **Codex（ChatGPT アプリ → Codex cloud → GitHub）** とする。自宅 PC を常時起動しなくても、外出先から記事化できる。

### 6.1 記事作成フロー

1. **質問**: 「質問〜」で始まる形式で疑問を投げる
   - 外出先: スマホの ChatGPT アプリから Codex にこのリポジトリを指定して指示（音声入力可）
   - 在宅時: PC 上の Claude Code / Codex にこのリポジトリで直接指示
2. **生成**: エージェントが Web 調査を行い、記事 HTML を生成。index.html も更新する
3. **確認**: 人間が差分を目視確認する（ガードレール）
   - 遠隔（Codex cloud）: 作業ブランチの **PR 差分**を GitHub 上で確認
   - ローカル: エージェントが作成したコミットと差分を `git show` などで確認
4. **公開**: PR をマージ（またはローカルで承認後に push）。GitHub Pages に公開される

### 6.2 コミット規則

- エージェントは依頼された作業のコミットまで行ってよい
- push は人間の明示承認後のみ行う
- エージェントは `main` へ直接プッシュしない。遠隔運用は作業ブランチ + PR、ローカルはコミット後に push せず止める
- 公開（`main` へのマージ）は人間が差分を確認してから行う
- コミットメッセージ: `add: <カテゴリ>/<スラッグ> <記事タイトル>` の形式
- 既存記事の修正は `fix:`、構成変更は `chore:` を接頭辞とする

### 6.3 記事の品質要件

- 回答は調査に基づき、**参照 URL（一次情報）を必ず記事末尾に記載**する
- 推測で書いた箇所は「推測」と明示する
- 質問の文脈（なぜ疑問に思ったか）を冒頭に残し、後から読み返したとき意図がわかるようにする

## 7. セキュリティ・機密情報の扱い

- 公開リポジトリで運用するため、**機密情報を含むファイルは Git 管理外**とする
  - 個人設定の JSON / YAML、API キー、ローカルパス情報などは `.gitignore` に登録
- エージェント向けの運用ルール・禁止事項は `AGENTS.md` に明文化する
  - 例: 「機密情報を記事に書かない」「push 前に必ず承認を得る」
- 人間による差分の目視確認を最終ガードレールとする

## 8. エージェント設定（AGENTS.md / CLAUDE.md）

両エージェント（Claude Code / Codex）から同じルールで運用できるようにする:

- `AGENTS.md`: 運用ルールの本体（記事テンプレート、命名規則、参照 URL 必須、push 承認制など）
- `CLAUDE.md`: `AGENTS.md` を参照する薄いラッパーとし、ルールの二重管理を避ける
- 記載する主なルール:
  - 記事は日本語で書く
  - 1 ページ 1 テーマ
  - 参照 URL を記事末尾に必ず記載
  - index.html の更新を忘れない
  - コミットはエージェントが行ってよい
  - push は人間の承認後のみ
  - ファイル削除は `cleanup.sh` への追記で行う

## 11. 非対象（やらないこと）

- ビルドツール・静的サイトジェネレータの導入（HTML 直書きを維持する）
- データベースやサーバサイド処理（静的ホスティングのみ）
- エージェントによる無人プッシュ（人間の承認を必須とする）
