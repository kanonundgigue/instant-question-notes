# AGENTS.md

このリポジトリは、技術的な疑問を静的 HTML の記事として蓄積する「instant-question-notes」です。Codex / Claude Code などのエージェントは、このファイルを運用ルールの本体として扱います。

## 基本方針

- 記事は日本語で書く
- 1 ページ 1 テーマにする
- 回答は一次情報の調査に基づいて書く
- 推測を書く場合は「推測」と明示する
- 機密情報、個人情報、未公開の認証情報を記事に含めない
- ビルドツールや静的サイトジェネレータを導入しない
- `main` ブランチへ直接プッシュしない（公開は人間の承認＝マージで行う）

## 依頼の省略ルール

このリポジトリでユーザーが技術的な疑問だけを送った場合、特に断りがなければ「その疑問を instant-question-notes の記事にする」依頼として扱います。

これは最優先の既定動作です。質問に対してその場で説明して終わるのではなく、まず記事作成フローを開始してください。短い回答で済む質問でも、公開ノートとして残せる内容なら記事にします。

このリポジトリ自身の使い方、運用ルール、公開方法、エージェントへの依頼方法に関する質問も記事化対象です。

例:

- `uv.lock はいつ更新される？`
- `GitHub Actions の permissions は何を指定すべき？`
- `Python の pathlib.Path と os.path はどう使い分ける？`
- `このプロジェクトの使い方は？`
- `Codex にどう依頼すれば記事になる？`

ただし、ユーザーが「説明だけ」「相談だけ」「記事にしないで」「差分を見たいだけ」などと明示した場合は、記事作成フローを開始せず、その指示を優先します。

記事化するか迷う場合は、記事化する側に倒します。記事化しない判断をするのは、ユーザーが明示的に記事化しない意図を示した場合、または機密情報・個人情報・未公開情報を含み公開ノートにできない場合に限ります。

## 記事作成フロー

1. 質問内容からカテゴリ名とスラッグを決める
2. 必要に応じて Web 調査を行い、一次情報を確認する
3. `notes/_template/article.html` を複製して `notes/<category>/<slug>.html` を作る
   - テンプレート内の `{{ }}` プレースホルダは**すべて**置き換える（残っているとそのまま表示される）
4. `index.html` の記事カードを追加し、日付降順に並べる
5. 成果物を人間が確認できる形にする
   - Codex cloud / 遠隔運用: 作業ブランチを切ってコミットし、PR を作成する
   - ローカル運用: エージェントがコミットまで行い、push せず差分とコミットを提示して止まる

## 記事ページの必須要素

- `<h1>`: 質問の主旨を表すタイトル
- メタ情報: 作成日、カテゴリ、元の質問文
- 本文: `<h2>` / `<h3>` で構造化する
- コード例: 必要な場合は `<pre><code>` で記載し、言語名を本文で明記する
- 参照: 記事末尾に一次情報の URL を列挙する
- ナビゲーション: トップページへの戻りリンクを置く

## 命名規則

| 対象 | 形式 | 例 |
|---|---|---|
| カテゴリ名 | 英小文字ケバブケース | `python`, `git`, `climate-data` |
| 記事ファイル名 | 英小文字ケバブケース + `.html` | `uv-lockfile-basics.html` |

## HTML / CSS ルール

- 記事固有の `<style>` は原則使わない
- 共通 CSS は `styles/main.css` のみを読み込む
- トップページから CSS を読む場合は `styles/main.css`
- 記事ページから CSS を読む場合は `../../styles/main.css`
- 画像を追加する場合は `images/` 配下に置く

## index.html 更新ルール

記事追加時は `index.html` の `.note-grid` にカードを追加します。

カードに含める情報:

- カテゴリラベル
- 記事タイトル
- 作成日
- 記事へのリンク

並び順は日付降順です。同日の記事が複数ある場合は、新しく追加した記事を上に置きます。

最初の記事を追加するときは、`index.html` の `.empty-state` ブロック（「まだ記事はありません」）を削除します。

## デプロイ対象

GitHub Pages で公開されるのは以下だけです（`.github/workflows/pages.yml` でステージングします）。

- `index.html`
- `styles/`
- `images/`
- `notes/`（ただし `notes/_template/` は除外）

`AGENTS.md` / `CLAUDE.md` / `plan.md` / `README.md` / `cleanup.sh` などの運用ドキュメントは公開されません。新しく公開したい種類のファイルを増やす場合は、ワークフローのステージング手順にも追記が必要です。

## 削除ルール

ファイル削除が必要な場合、直接削除せず `cleanup.sh` に削除コマンドを追記して人間の確認を待ちます。

## コミットルール

- 新規記事: `add: <カテゴリ>/<スラッグ> <記事タイトル>`
- 既存記事修正: `fix: <対象> <内容>`
- 構成変更: `chore: <内容>`
- エージェントは依頼された作業のコミットまで行ってよい
- push は人間の明示承認後のみ行う
- `main` へ直接プッシュしない。遠隔運用では作業ブランチ + PR、ローカルではコミット後に push せず止める
- 公開（`main` へのマージ）は人間が PR の差分を確認してから行う

## Codex で外出先から使う場合

このリポジトリは Codex の `AGENTS.md` 規約に対応しています。スマホの ChatGPT アプリから Codex に質問を渡すと、Codex cloud がこのリポジトリに対して作業します。

1. ChatGPT アプリで Codex を開き、このリポジトリを選ぶ
2. 「質問〜 ◯◯について記事にして」と渡す（音声入力可）
3. Codex がこのファイルのルールに従って記事を作り、**作業ブランチに PR を作成**する
4. 人間が GitHub 上で PR の差分を確認し、問題なければマージする
5. `main` へのマージで GitHub Pages が公開される

Codex は自宅 PC を起動していなくても cloud 上で動くため、外出先からの記事化に向く。
