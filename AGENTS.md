# AGENTS.md

このリポジトリは、疑問や調べたことを静的 HTML の記事として蓄積する「instant-question-notes」です。Codex / Claude Code などのエージェントは、このファイルを運用ルールの本体として扱います。

## 基本方針

- 記事は日本語で書く
- 1 ページ 1 テーマにする
- 調査が必要な回答は一次情報の確認に基づいて書く
- 推測を書く場合は「推測」と明示する
- 音楽関係の質問は、初心者向けの入門解説ではなく、プロが知識を整理するための質問として扱う
- 機密情報、個人情報、未公開の認証情報を記事に含めない
- ビルドツールや静的サイトジェネレータを導入しない
- リポジトリは所有者のみが更新できる設定にし、公開は `main` ブランチへの直接 push で行う
- 作業ブランチや PR は、ユーザーが明示的に求めた場合だけ使う

## 依頼の省略ルール

このリポジトリでユーザーが疑問や調べたいことだけを送った場合、特に断りがなければ「その内容を instant-question-notes の記事にする」依頼として扱います。

これは最優先の既定動作です。質問に対してその場で説明して終わるのではなく、まず記事作成フローを開始してください。短い回答で済む質問でも、公開ノートとして残せる内容なら記事にします。

このリポジトリ自身の使い方、運用ルール、公開方法、エージェントへの依頼方法に関する質問も記事化対象です。

例:

- `uv.lock はいつ更新される？`
- `GitHub Actions の permissions は何を指定すべき？`
- `Python の pathlib.Path と os.path はどう使い分ける？`
- `引っ越し前に住所変更が必要なサービスは？`
- `このプロジェクトの使い方は？`
- `Codex にどう依頼すれば記事になる？`

ただし、ユーザーが「説明だけ」「相談だけ」「記事にしないで」「差分を見たいだけ」などと明示した場合は、記事作成フローを開始せず、その指示を優先します。

記事化するか迷う場合は、記事化する側に倒します。記事化しない判断をするのは、ユーザーが明示的に記事化しない意図を示した場合、または機密情報・個人情報・未公開情報を含み公開ノートにできない場合に限ります。

## 記事作成フロー

1. 質問内容からカテゴリ名とスラッグを決める
2. 必要に応じて Web 調査を行い、一次情報を確認する
3. `notes/_template/article.html` を複製して `notes/<category>/<slug>.html` を作る
   - テンプレート内の `{{ }}` プレースホルダは**すべて**置き換える（残っているとそのまま表示される）
4. `index.html` の新着セクション、カテゴリ導線、カテゴリ別セクションの記事カードを追加または更新する
5. 記事カードは、更新日がある記事は更新日、更新日がない記事は作成日を基準に日付降順で並べる
6. 成果物を人間が確認できる形にする
   - ローカル運用: エージェントは記事ファイルと `index.html` を更新し、差分を提示して止まる。コミットはしない。`commitして` と指示されたときにコミットする（それまでは追加修正を続けられる）
   - Codex cloud / 遠隔運用: 作業ブランチや PR は作らず、ユーザーが `pushして` / `公開して` などと明示した場合に `main` へ直接 push する

## 保護記事（パスワード付き）

一部の記事だけパスワードがないと中身を読めないようにできます。普通の記事は今まで通り公開のままで、保護したい記事だけこの仕組みを使います。

仕組みは「コンテンツを暗号化して公開し、ブラウザ内でパスワード復号する」方式です。GitHub Pages はサーバー側処理を持たないため本物のログイン認証はできませんが、この方式なら**公開されるのは暗号文だけ**になります。外部ライブラリやビルドツールは使わず、ブラウザ標準の Web Crypto API（AES-GCM + PBKDF2）だけで実装しています。

### いつ保護するか（既定は公開）

- 記事は**既定で公開**。通常どおり作成・コミットする。
- **保護するのは、ユーザーがコミット前に「この記事は保護して」と明示的に指示したときだけ。** 指示がなければ公開でよい。
- 保護記事はすべて**共通の 1 つのパスワード**で暗号化する。実際のパスワード文字列はローカル専用メモ `PROTECTED_NOTES.local.md`（`.gitignore` 済み）に記載されている。エージェントは、その値を会話で渡された場合のみ暗号化に使う。リポジトリやコミットにパスワードを書かない。

### 作成フロー

1. プレビューサーバを起動し、`tools/encrypt.html` を **localhost 経由**で開く
   - Web Crypto はセキュアコンテキスト（https / localhost）でしか動かない。`file://` で直接開くと暗号化できない
2. 「公開してよいタイトル」「記事本文の HTML（`<h1>` から）」「パスワード」を入力して暗号化する
   - 本文 HTML は通常記事と同じ構造（`<h1>`・`.article-meta`・各 `<section>`）で書く
3. 生成された HTML 全文をコピーし、`notes/<category>/<slug>.html` として保存する
   - 雛形は `notes/_template/article-protected.html`。ツールが同じ構造のファイルを生成するので、通常はツール出力をそのまま保存すればよい
4. 保護領域の入口一覧 `notes/private/index.html` にカードを追加する（トップヘッダー右上の「🔒 保護領域」リンクの遷移先）。ロックバッジ（🔒）を付け、要約は中身が分からない一般的な文言にする
   - トップページ `index.html` の通常セクションにも載せるかは任意。載せる場合も同様にバッジを付ける

### 注意点（必ず守る）

- **平文（本文の中身）は絶対にコミットしない。** リポジトリにも公開ページにも残さない。下書きを残す場合は `_drafts/` か `*.draft.html`（どちらも `.gitignore` 済み）に置く
- 安全性は**パスワードの強さがすべて。** 公開リポジトリでは暗号文を誰でも取得できるため、長く推測されにくいパスフレーズを使う
- **公開される情報**: 記事タイトル・要約・URL・ページ `<title>`。これらに秘密を書かない（本文だけが暗号化対象）
- このリポジトリの方針では全保護記事で共通パスワードを使う。利便性を優先した判断で、共通パスワードが 1 度漏れると全保護記事が読まれる点は許容する。だからこそ十分に強いパスフレーズにする
- 機密情報・個人情報・未公開の認証情報は、暗号化していても公開ノートには載せない原則を維持する

## 記事ページの必須要素

- `<h1>`: 質問の主旨を表すタイトル
- メタ情報: 作成日、後日修正時の更新日、カテゴリ、元の質問文
- 本文: `<h2>` / `<h3>` で構造化する
- コード例: 必要な場合は `<pre><code>` で記載し、言語名を本文で明記する
- 参照: 記事末尾に一次情報の URL を列挙する
- ナビゲーション: トップページへの戻りリンクを置く

## 既存記事の更新ルール

既存記事を後日修正する場合は、記事ページのメタ情報に更新日を追加または更新します。

- 作成日は初回作成日として変更しない
- 本文、参照 URL、運用方針、結論など、読者の理解に影響する内容を直した場合は `更新日: <time datetime="YYYY-MM-DD">YYYY-MM-DD</time>` を追加する
- すでに更新日がある記事を再修正する場合は、最新の修正日に置き換える
- 表記ゆれ、誤字、HTML の整形だけで内容が変わらない修正では、更新日は必須にしない
- `index.html` の記事カードにも更新日を表示し、作成日と更新日が一覧から分かるようにする

## HTML / CSS ルール

- 記事固有の `<style>` は原則使わない
- 共通 CSS は `styles/main.css` のみを読み込む
- トップページから CSS を読む場合は `styles/main.css`
- 記事ページから CSS を読む場合は `../../styles/main.css`
- 画像を追加する場合は `images/` 配下に置く
- テーブルを作る場合はスマホ幅で読める設計にする
  - まず列数を減らせないか検討し、原則 2-3 列程度に収める
  - 横に長い比較は、複数の小さな表、箇条書き、またはカード状の構造へ分ける
  - 表を使う場合は `<table class="mobile-table">` とし、各 `<td>` に対応する見出しを `data-label` で入れる
  - 狭い画面で横スクロールを前提にした幅指定、長い非改行テキスト、列数の多い表を作らない

## index.html 更新ルール

記事追加または既存記事更新時は、`index.html` だけで記事を探せるように更新します。トップページはページ内アンカーで移動する「1 枚看板」として扱い、カテゴリ別ページへの遷移を主要導線にしません。

更新対象:

- ヘッダーまたはカテゴリ一覧のページ内アンカー
- 新着セクション
- 該当カテゴリのカテゴリ別セクション

記事カードは `index.html` の `.note-grid` に追加または更新します。

カードに含める情報:

- カテゴリラベル
- 記事タイトル
- 作成日
- 更新日（既存記事を実質的に修正した場合）
- 記事へのリンク

並び順は、更新日がある記事は更新日、更新日がない記事は作成日を基準にした日付降順です。同日の記事が複数ある場合は、新しく追加または更新した記事を上に置きます。

最初の記事を追加するときは、`index.html` の `.empty-state` ブロック（「まだ記事はありません」）を削除します。

## カテゴリページの扱い

既存の `notes/<category>/index.html` は、過去の URL 互換のため残します。ただし、新しい記事追加時にカテゴリページを追加または更新することは必須ではありません。

新しいカテゴリを作る場合は、`index.html` にページ内アンカー付きのカテゴリ導線とカテゴリ別セクションを追加します。トップページからカテゴリページへリンクさせる構成には戻しません。

## デプロイ対象

GitHub Pages で公開されるのは以下だけです（`.github/workflows/pages.yml` でステージングします）。

- `index.html`
- `styles/`
- `scripts/`
- `images/`
- `notes/`（ただし `notes/_template/` は除外）

`AGENTS.md` / `CLAUDE.md` / `plan.md` / `README.md` / `cleanup.sh` などの運用ドキュメントは公開されません。`tools/`（暗号化ツール `encrypt.html` を含む）も意図的に配信対象外で、ローカル専用です。新しく公開したい種類のファイルを増やす場合は、ワークフローのステージング手順にも追記が必要です。

## GitHub リポジトリ設定

このリポジトリは個人用ノートとして、所有者だけが更新できる状態で運用します。

- GitHub の `Settings` > `Collaborators & teams` で、所有者以外に write / maintain / admin 権限を持つユーザーやチームを置かない
- public repository でも、collaborator でない第三者は通常 push できない。第三者の提案は fork や pull request になる
- `main` ブランチには直接 push できる状態にする
- branch protection rule / ruleset で `Require a pull request before merging`、`Lock branch`、`Restrict updates` などが有効になっていて直接 push を妨げる場合は無効化する
- push できる人を制限する branch protection / ruleset を使う場合は、所有者または信頼済みの自動化だけを許可する

## 削除ルール

ファイル削除が必要な場合、直接削除せず `cleanup.sh` に削除コマンドを追記して人間の確認を待ちます。

## コミットルール

- 新規記事: `add: <カテゴリ>/<スラッグ> <記事タイトル>`
- 既存記事修正: `fix: <対象> <内容>`
- 構成変更: `chore: <内容>`
- ローカル運用では、エージェントは指示されない限りコミットしない。作業後は差分を提示して止まり、`commitして` と指示されたときにコミットする。記事に追加修正を入れたい場合に備え、コミット前の状態で止めておく
- 遠隔運用（Codex cloud）では、ユーザーが `pushして` / `公開して` などと明示した場合に `main` へ直接 push してよい
- push は人間の明示承認後のみ行う
- 作業ブランチや PR は、ユーザーが明示的に求めた場合だけ使う
- 公開は `main` への直接 push によって行う

## Codex で外出先から使う場合

このリポジトリは Codex の `AGENTS.md` 規約に対応しています。スマホの ChatGPT アプリから Codex に質問を渡すと、Codex cloud がこのリポジトリに対して作業します。

1. ChatGPT アプリで Codex を開き、このリポジトリを選ぶ
2. 「質問〜 ◯◯について記事にして」と渡す（音声入力可）
3. Codex がこのファイルのルールに従って記事を作る
4. 公開まで任せる場合は「pushして」または「公開して」と明示する
5. Codex が `main` へ直接 push し、GitHub Pages が公開される

Codex は自宅 PC を起動していなくても cloud 上で動くため、外出先からの記事化に向く。
